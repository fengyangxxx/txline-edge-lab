## Live Validation Contract

The live contract check feeds a TxLINE-shaped SSE packet into
`normalizeTxlineMessage`, then into the same signal engine used by the browser.
The generated proof must show:

- `mode = live-validation-ready`;
- `MessageId = odds-msg-9002`;
- odds validation path `/api/odds/validation?messageId=odds-msg-9002`;
- score validation path `/api/scores/stat-validation?...`;
- TxOracle instruction names `validate_odds` and `validate_stat`;
- paper action `paper-long` only after the execution gate passes.
- package proof includes `bin/txline-edge-lab.mjs` and a SHA256 for the local npm
  tarball;
- Solana proof envelope includes a `txline-edge-lab:<sha256>` memo payload and
  `broadcast_status = not_broadcast`.

## Competitive Positioning

The strongest public competitors are proof-first. TxLINE Edge Lab responds with:

- deterministic replay SHA rather than a screenshot-only demo;
- live-validation-contract artifact rather than a docs-only claim;
- CLI/npm package proof instead of a hosted-dashboard-only surface;
- honest Solana memo payload instead of pretending replay data has already been
  broadcast on-chain;
- CLV proxy and capped paper risk instead of raw PnL claims;
- anti-chase execution gate that keeps score-driven moves visible but blocks
  unsafe auto-execution;
- session-only credential handling and no wallet or wager requirement for review.

## Honest Limits

- Replay digest is not live on-chain attestation.
- The Solana Memo payload is not a transaction signature; it is ready for a
  reviewer wallet to sign if desired.
- Public repo does not bundle live TxLINE credentials or real match packets.
- Product is analytics and paper execution only; it intentionally avoids real
  betting automation, custody, deposits, and reviewer wallet requirements.
